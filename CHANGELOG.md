# Changelog

## [Unreleased]

Nothing is published yet, so the entries below describe `v0.1` as a whole rather
than deltas against a released version.

### Added

- `#[derive(Layered)]`, implementing `flag > env > file > default` precedence
  driven by clap's `ValueSource`. A flag the user explicitly types beats every
  lower layer **even when the typed value equals the built-in default**, which is
  the crate's core correctness claim.
- `Layered::layered()` — loads from the real process arguments and environment.
  Like `clap::Parser::parse`, it defers to clap for CLI errors and `--help`.
- `Layered::layered_from(args, env)` — the same engine with the arguments and
  environment injected, so configuration is testable and parallel-safe.
- `Env`, an explicit environment snapshot (`Env::from_system`, `Env::empty`,
  `FromIterator`).
- Real TOML support via `toml` and `serde`. Any `serde::Deserialize` field type
  works, including `u16`, `bool`, `String`, `Vec<T>`, `Option<T>` and enums; no
  `FromStr` bound is required. Nested tables deserialize into nested structs.
- Source-attributed errors: `LayeredError::{Cli, Io, Parse, Invalid}` and
  `SourceLayer`. A bad value names its layer and, for config files, its line —
  e.g. `invalid value 'foo' for 'port' — from config.toml, line 12`.
- Per-field markers `no_cli`, `no_env` and `no_file`. Malformed, misspelled or
  duplicated `#[layered(...)]` options are now compile errors with help text,
  covered by `trybuild` compile-fail tests.
- Precedence-matrix test suite covering `(flag | env | file | none)` against
  `u16`, `bool`, `String`, `Vec<T>`, `Option<T>` and enums.
- Property-based tests (`proptest`) over generated layer stacks, asserting the
  rule rather than enumerated cases: the highest-priority layer that is set wins,
  a marker removes its layer entirely, a lower layer cannot disturb a higher one,
  and strings survive both the file and environment layers verbatim.
- End-to-end tests that run a real binary, covering `layered()` itself — the
  process argv/env it reads and the `clap::Error::exit` path are unreachable
  in-process.
- clap's own `#[arg(env = "...")]` composes: a value clap read from the
  environment counts as explicit and keeps beating the config-file layer.
- `#[command(flatten)]` and subcommand fields are passed through untouched
  rather than being resolved as scalars.

### Fixed

Found while hardening against edge cases:

- **Raw-identifier fields panicked.** A field written `r#type` is the argument id
  `type` to clap; passing the raw spelling to `ArgMatches::value_source` queried
  an id clap never registered, which panics in debug builds.
- **Structs generated by a `macro_rules!` macro were rejected.** A substituted
  `$file:literal` arrives wrapped in an invisible `Expr::Group`, so a valid
  string literal was refused with "`file` must be a string literal".
- **Config-file columns counted bytes, not characters**, so a file containing any
  multi-byte text reported a column that disagreed with `toml`'s own diagnostics
  and with every other tool.
- A span pointing past the end of a file, or into the middle of a character,
  could panic while rendering the error position.
- `env_prefix` values that cannot name an environment variable (empty, `my-app`,
  a leading digit) are now compile errors instead of building variables like
  `MY-APP_PORT` that no shell can set.
- `#[layered(file = "")]` is a compile error rather than a layer that silently
  never applies.
- The examples printed errors with `expect`, which renders the `Debug`
  representation and discards the source-attributed message.

Against the unreleased scaffolding that preceded it:

- **Precedence was inverted.** The generated code applied the environment and
  file layers *over* the parsed CLI value, so `--port 9999` lost to both
  `PORT=1234` and a config file. `ValueSource` was never consulted.
- **`no_cli` / `no_env` / `no_file` were silently ignored.** Field attributes
  were parsed as `MetaNameValue`, so a bare marker failed to parse and the error
  was discarded — a field marked `no_env` still read the environment. Unknown and
  misspelled options were likewise ignored rather than rejected.
- **Bad values were silently swallowed.** Parse failures from the environment and
  file layers fell back to the default instead of erroring, so a typo'd
  `PORT=banana` quietly started the app on the wrong port.
- **The TOML parser was not a TOML parser.** It split lines on `=`, ignoring
  `[section]` headers (so `[a].port` and `[b].port` collided), arrays, inline
  tables and quoting.
- **Environment variable names were not uppercased**, so the documented
  `MYAPP_PORT` in fact had to be written `MYAPP_port`.
- The environment layer no longer runs without an `env_prefix`; previously a
  field named `path` would read the ambient `PATH`.
- The config file is read and parsed **once** per load rather than once per
  field.

### Changed

- Integration tests moved into `crates/runtime/tests/`, where cargo discovers
  them automatically. The previous `tests/` package registered targets by hand,
  and the precedence-matrix suite was never listed — so it never compiled or ran.
- `cargo test` at the workspace root now runs every crate's tests; the previous
  `default-members` setting limited it to the runtime crate.
- `Env::from_iter` accepts any `Into<String>` pair, so `[("MYAPP_PORT", "8080")]`
  works without `to_string()` on both halves.
- `SourceLayer` lost its `CliFlag` and `Default` variants: nothing could
  construct them, because a malformed CLI value is reported by clap as
  `LayeredError::Cli`. The enum is `#[non_exhaustive]`, so `--dump-config` can
  reintroduce them without a breaking change.
- A `[workspace.lints]` table turns on `clippy::pedantic` and `clippy::cargo`;
  library-only strictness (`forbid(unsafe_code)`, `deny(missing_docs)`, no
  `unwrap`/`panic`) sits at the crate roots so it does not fire on test code.

### Security

- `cargo-deny` runs in CI on every push and weekly on a schedule, since an
  advisory can be published against an unchanged `Cargo.lock`. `deny.toml`
  allows only the `MIT`, `Apache-2.0` and `Unicode-3.0` licences the dependency
  tree actually resolves to, requires every crate to come from crates.io, and
  denies wildcard versions and duplicate versions of shipped crates. It has no
  exceptions.

### Dependencies

- `toml` 0.8 -> 1. The runtime crate held the tree on `toml` 0.8 while `trybuild`
  pulled `toml` 1, so `toml`, `serde_spanned`, `toml_datetime` and `winnow` were
  each present at two versions. No `toml` type appears in this crate's public
  API, so the change is internal. The dependency graph drops from 86 crates to
  80.

### Known limitations

- A field with **no default and no `Option`** cannot be filled from the file or
  environment: clap enforces required-ness while parsing, before any lower layer
  is consulted. Give the field a default or make it `Option<T>`. Pinned by a test.
- Config-file keys with no matching field are **ignored**, not rejected, so a
  typo in a config file does nothing silently. A strict mode is a candidate for a
  later version.
- Sequences from the environment must be TOML arrays (`MYAPP_TAGS='["a","b"]'`);
  clap's `value_delimiter` applies only to the command line.
